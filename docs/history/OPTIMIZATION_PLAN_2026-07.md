# CF-Navs 全项目优化计划（2026-07）

## 目标与优先级

本轮优化按以下顺序决策：

1. 本地性能：减少主线程工作、重复序列化、重复渲染和不必要的浏览器存储。
2. 云端请求：减少可合并、可复用或无业务价值的 Worker、D1、KV 与第三方请求。
3. 安全防护：限制公开入口滥用，收紧输入、响应与浏览器安全边界。
4. 可维护性：按路由编排、纯验证、限流存储、客户端传输等责任拆分，不把新逻辑继续堆入大文件。

所有阶段必须保持分类、书签、设置、导入、认证、公开模式和多设备数据版本一致性不变。

## 当前基线

- 前端：Svelte 4 + TypeScript + Vite；主入口 `App.svelte` 约 26 KB，数据编排已下沉到 `dataService.ts`。
- Worker：Hono + D1 + KV；公开聚合和后台聚合已有短时缓存及数据版本机制。
- 当前生产数据：11 个分类、348 个书签。
- 登录态聚合响应约 38 KB 网络传输；客户端登录态快照约 159 KB。
- 生产首页完整滚动仍会产生大量书签图标 Worker 请求；不得通过扩大 Cache Storage 或嵌入 `icon_blob` 回退到旧问题。
- 安全响应已有 CSP、`X-Frame-Options`、`nosniff`、Permissions Policy。
- 登录已有 IP 失败限流、PBKDF2、KV Session、短时 Worker 内存 Session 缓存。
- 测试基线：42 个测试文件、249 项单元测试；生产回归脚本覆盖首页、后台、认证、右键编辑和安全探针。

## 审计发现与风险排序

### P0：公开错误上报入口缺少服务端资源边界

现状：

- `/api/error-report` 无需认证，当前接受任意 JSON 大小、任意数组长度，并逐项写入 `console.error`。
- 客户端只做实例内批处理；恶意请求、脚本循环或多个标签页不受该限制。
- 无效 payload 也可能返回成功，无法区分拒绝、截断和有效接收。

风险：

- 可被用于放大 Worker 调用与日志写入。
- 大请求体和大数组会增加解析、循环和日志成本。
- 重复错误会产生无业务价值的云端请求。

计划：

- 新建独立的错误上报验证/归一化 helper，路由只负责编排。
- 限制 Content-Length、实际 JSON 体积、批次条目数、单字段长度和允许字段类型。
- 对公开错误上报增加独立 IP 窗口限流；存储复用 KV，但使用独立 key 前缀。
- 客户端增加同一错误短时去重，并降低批次发送上限。
- 明确返回接收数量；超限使用 HTTP 413/429，避免继续处理。

验收：

- 正常错误批次仍可上报。
- 超大请求、超大批次、畸形字段和高频请求被稳定拒绝或截断。
- helper 和路由均有单元测试。
- 不增加正常页面启动请求；没有错误时仍为 0 次错误上报。

### P1：客户端聚合快照模块存在重复实现

现状：

- `publicDataCache.ts` 与 `adminDataCache.ts` 重复实现序列化、localStorage/Cache Storage 回退、容量限制和清理。
- 两份实现以后容易出现容量、清理或损坏处理不一致。

计划：

- 抽取类型无关的单快照存储 helper。
- 公开和登录态模块只保留 key 作用域、结构校验和会话隔离。
- 保持 localStorage 优先、Cache Storage 回退、1.5 MB 上限和版本校验语义不变。

验收：

- 浏览器存储数量和大小不增长。
- 现有快照兼容读取；损坏和超限测试继续通过。
- 弱网恢复性能不退化。

### P1：图标请求与失败重试审计

现状：

- 348 个书签的完整滚动仍可能触发约 200 级别的同源图标请求。
- 当前已避免把所有图标写入 Cache Storage，且通过懒加载、D1 blob、edge cache 和临时 fallback 控制成本。

计划：

- 先用生产 `perf:audit` 记录图标请求来源、重复 URL、fallback 比例和 edge 命中线索。
- 只优化真实重复请求或同一会话内的失败重试；不恢复批量 Cache Storage 图标缓存。
- 优先考虑稳定 URL 去重、失败冷却和已知 fallback 的会话级复用。

验收：

- 相同浏览路径的 Worker 图标请求数下降或不增加。
- 破图为 0，浏览器存储保持在性能契约范围内。

### P1：API GET 并发复用审计

现状：

- API 客户端对所有请求直接 `fetch`，没有通用的同请求并发复用。
- 当前启动编排通常只发一个版本请求，因此不能在没有证据时加入复杂缓存层。

计划：

- 在浏览器网络记录中定位真实重复的只读请求。
- 仅对同一时刻、同 URL、同认证作用域的幂等 GET 做 in-flight promise 复用。
- 不缓存完成结果，不改变 no-cache、数据版本或登录失效语义。

验收：

- 并发重复请求合并，顺序请求仍正常到达服务端。
- 401 清理、本地快照和强制刷新不受影响。

### P2：大组件继续按责任拆分

候选：

- `App.svelte`：启动/登录门禁与 CRUD/弹窗编排仍可进一步拆分 controller。
- `Home.svelte`：搜索派生、滚动高亮和导航释放计时可拆为纯状态 helper。
- `BookmarkEditModal.svelte`：Iconify 搜索计时与表单 UI 可继续解耦。

原则：

- 只在有测试保护的责任边界拆分。
- 每轮只移动一类职责，保持 DOM、API 调用和交互行为不变。
- 不为了减少文件长度制造跨模块隐式状态。

## 二次评估结论

- 不先重写 App 或 Home：文件虽大，但当前功能测试稳定；直接大拆分的风险高于短期性能收益。
- 不先加入通用 GET 缓存：现有数据版本和 no-cache 语义明确，没有生产重复请求证据前容易造成一致性问题。
- 不扩大 Service Worker 图标缓存：此前已证明会导致多 MB 浏览器存储增长。
- 第一阶段选择错误上报链路：边界清晰、可独立测试，同时覆盖安全、云端请求和解耦三项目标。
- 第二阶段选择快照存储 helper：属于行为保持型维护优化，可降低后续缓存策略漂移风险。
- 图标与 GET 请求优化延后到生产指标之后，避免为“看起来可能重复”的请求引入错误缓存。

## 阶段 1 实现说明

- 新增 `worker/lib/errorReportValidation.ts`：纯函数负责 body、批次和字段边界以及输入归一化。
- 新增 `worker/lib/errorReportRateLimit.ts`：使用 D1 原子 UPSERT 作为跨 Worker isolate 的计数真源，已封禁来源再由 isolate 内存快速拒绝，按 IP 控制每分钟最多 12 次有效上报。生产探针证明 KV 最终一致性和单 isolate 内存都不能独立提供全局精确限流，因此最终方案不再依赖 KV 计数。
- 路由先拒绝超大或无效请求，再消耗限流配额，避免为垃圾 payload 产生不必要 KV 写入。
- 客户端同一错误指纹 60 秒内只入队一次，每分钟最多发送 6 个批次。
- 正常页面没有错误时仍不会调用 `/api/error-report`。
