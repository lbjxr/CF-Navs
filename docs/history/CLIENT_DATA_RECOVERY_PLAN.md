# 客户端首页数据恢复计划

## 目标

在网络延迟、超时或临时离线时，已访问过站点的浏览器应尽快恢复首页；同时继续执行远端版本校验，避免长期展示过期数据，并限制浏览器快照对客户端存储的占用。

## 加载策略

### 首次访问

浏览器没有可用快照时，仍需请求 `/api/public/data` 或 `/api/admin/data`。首次数据成功返回后写入一份带版本号的聚合快照。首次访问没有旧数据可恢复，因此网络失败时保留现有错误提示。

### 后续刷新

1. 恢复本地登录会话。
2. 读取与当前站点、当前登录会话匹配的聚合快照。
3. 快照通过结构校验后立即写入 Store 并解除首页启动遮罩。
4. 后台请求 `/api/data/version` 验证一致性。
5. 版本相同：保留快照，不拉取完整聚合数据。
6. 版本不同：拉取最新聚合数据、更新 Store，并覆盖本地快照。
7. 版本请求失败：继续显示快照，通过非阻断小弹窗提示用户刷新或检查网络。

该策略属于 stale-while-revalidate，但数据版本检查仍是每次启动的必要步骤，不增加轮询和额外启动请求。

## 一致性约束

- 本地快照不能替代 `/api/data/version`，只用于提前恢复可见内容。
- 分类、书签、设置、排序和导入成功后，继续同步更新内存 Store 与本地快照。
- 登录态快照按站点 origin 和当前 session 隔离；session 变化时清理旧登录快照。
- 公开模式关闭、登录失效或接口明确返回未授权时，不允许继续用旧快照绕过权限状态。
- 弱网失败只触发降级提示；明确的权限或公开模式变化仍按服务端结果清理缓存并切换登录门禁。

## 存储约束

- 公开快照和登录态快照各自最多保留一份。
- 单份序列化快照上限为 1.5 MB；超过上限时清理该类快照，不继续占用客户端空间。
- 优先写入 localStorage，写入成功后删除同类 Cache Storage 副本。
- localStorage 不可用、隐私模式限制或配额不足时，才回退 Cache Storage。
- 聚合数据继续排除大体积 `icon_blob`；图标使用独立、受控的缓存路径。

## 验证计划

- 单元测试：快照立即应用、版本一致短路、版本变化同步、弱网保留快照并提示、无快照错误路径、存储上限与单副本行为。
- 构建检查：`npm run type-check`、`npm test`、`npm run build`、`git diff --check`。
- 浏览器验证：使用独立 Profile 或无头 Chrome，完成一次在线加载以建立快照，再阻断/延迟数据版本请求并刷新，确认首页先于远端请求完成展示、小弹窗出现、无页面异常，并检查快照数量和占用。
